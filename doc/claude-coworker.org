*** Background: what the ACL mask is

A POSIX ACL on a file has this structure:

#+begin_example
user:≡rwx          # owning user
user:claude:rwx    # named user entry  ← what setfacl sets
group:≡r--         # owning group
mask:≡rwx          # THE MASK
other:≡r--         # others
#+end_example

The *mask* is a cap. The effective permission of any named user or
group entry is:

#+begin_example
effective = entry_permissions AND mask
#+end_example

The named user entry (=user:claude:rwx=) is what you set with =setfacl=.
But what =claude= can /actually do/ is that entry ANDed with the mask.
If the mask is =rw-=, then even if the entry says =rwx=, the effective
permission is =rw-=.

*** Where the mask comes from on a newly created file

When a process creates a new file, the kernel computes the base
permissions as:

#+begin_example
file_permissions = requested_mode AND (NOT umask)
#+end_example

For a regular file, the requested mode is typically =0666=. With
=umask 022=:

#+begin_example
0666 AND NOT(022) = 0666 AND 0755 = 0644  →  rw-r--r--
#+end_example

With =umask 077=:

#+begin_example
0666 AND NOT(077) = 0666 AND 0700 = 0600  →  rw-------
#+end_example

When a file is created inside a directory that has a *default ACL*,
the kernel initialises the new file's ACL mask from the base
permissions — specifically from the group permission bits of the
computed mode. With =umask 077= those bits are =---=, so the mask
becomes =---=.

*** The consequence

Suppose =owner= has =umask 077= and creates a new file inside =$INFRA=:

#+begin_src sh
touch $INFRA/newfile
#+end_src

The resulting ACL is:

#+begin_example
user:≡rw-           # owner can read/write
user:claude:rw-     # setfacl entry says rw-
group:≡---
mask:≡---           # ← derived from umask 077, group bits = 000
other:≡---
#+end_example

Effective permission for =claude=:

#+begin_example
rw- AND --- = ---
#+end_example

=claude= can do nothing, despite the explicit ACL entry granting =rw-=.
The mask silently overrides it. =getfacl= will expose this with an
annotation:

#+begin_example
user:claude:rw-         #effective:---
#+end_example

The =#effective:---= annotation is the diagnostic signal that the mask
is clipping the named user entry.

*** The fix

Ensure =owner= uses =umask 022= when writing to =$INFRA=, so the mask is
initialised to =rw-=:

#+begin_src sh
umask 022
touch $INFRA/newfile
# mask is now rw-, claude's effective permission is rw-
#+end_src

Or re-apply =setfacl= after the fact. This resets the mask to the
union of all named entry permissions, restoring =claude='s effective
permissions regardless of what umask produced:

#+begin_src sh
sudo setfacl -R -m u:claude:rwX "$INFRA"
#+end_src

To make =umask 022= the default for all shell sessions writing to
=$INFRA=, add it to =owner='s shell initialisation file (=~/.bashrc=,
=~/.profile=, or equivalent).

** TODO Configure User Namespace UID/GID Mapping for =owner=

=guix shell -C= uses unprivileged Linux user namespaces. The kernel
requires that the invoking user (=owner=) have subordinate UID and GID
ranges declared in =/etc/subuid= and =/etc/subgid=. These ranges allow
=owner= to map UIDs inside the namespace without requiring root.

Check whether entries already exist:

#+begin_src sh
grep "^owner:" /etc/subuid
grep "^owner:" /etc/subgid
#+end_src

If absent, add them:

#+begin_src sh
sudo usermod --add-subuids 100000-165535 owner
sudo usermod --add-subgids 100000-165535 owner
#+end_src

Verify:

#+begin_src sh
cat /etc/subuid
# owner:100000:65536

cat /etc/subgid
# owner:100000:65536
#+end_src

The range =100000-165535= (65536 entries) is the Debian convention. It
must not overlap with any existing system UIDs.

** TODO Create the XDG Runtime Directory for =claude=

The container command references =/run/user/$UID=. Since =claude= is a
system user with no login session, the =systemd-logind= daemon never
creates this directory automatically. It must be created manually:

#+begin_src sh
sudo mkdir -p /run/user/${CLAUDE_UID}
sudo chown claude:claude /run/user/${CLAUDE_UID}
sudo chmod 700 /run/user/${CLAUDE_UID}

# Verify
ls -ld /run/user/${CLAUDE_UID}
# drwx------ 2 claude claude ... /run/user/999
#+end_src

Note: this directory lives on a =tmpfs= and is lost on reboot. If the
container is started after a reboot, re-run the above three commands, or
add them to a systemd =tmpfiles.d= rule for persistence:

#+begin_src sh
# /etc/tmpfiles.d/claude-runtime.conf
echo "d /run/user/${CLAUDE_UID} 0700 claude claude -" \
  | sudo tee /etc/tmpfiles.d/claude-runtime.conf

# Apply immediately without rebooting
sudo systemd-tmpfiles --create /etc/tmpfiles.d/claude-runtime.conf
#+end_src

** TODO Configure =sudo= to Allow =owner= to Invoke the Container as =claude=

=claude= has =/usr/sbin/nologin= as its shell, so it cannot be switched
to with =su= or =su -=. The correct mechanism is =sudo -u claude=.

Determine the exact path to =guix=:

#+begin_src sh
which guix
# e.g. /usr/local/bin/guix
#+end_src

Add a sudoers rule granting =owner= the right to run =guix= as =claude=
without a password:

#+begin_src sh
sudo visudo -f /etc/sudoers.d/claude-container
#+end_src

Insert:

#+begin_src text
# Allow owner to start the guix container as claude
owner ALL=(claude) NOPASSWD: /usr/local/bin/guix
#+end_src

Adjust the path if =guix= is installed elsewhere (e.g. =/usr/bin/guix=).
=visudo= validates the syntax before saving; a syntax error will be
caught and rejected.

Verify:

#+begin_src sh
sudo -u claude -l
# Should list: (claude) NOPASSWD: /usr/local/bin/guix
#+end_src
